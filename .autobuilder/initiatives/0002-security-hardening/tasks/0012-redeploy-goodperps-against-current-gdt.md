## Context

`integration-results.md` (from task 0009) shows GoodPerps as
**FAIL ŌĆö vault.deposit reverted**. Diagnosis (this iteration, on-chain):

| Field                              | Value                                          |
|-----------------------------------|------------------------------------------------|
| `VAULT` (env: `MarginVault`)       | `0xf4b146fba71f41e0592668ffbf264f1d186b2ca8` |
| `VAULT.collateral()` (immutable)   | `0x36C02dA8a0983159322a80FFE9F24b1acfF8B570` (DEAD ŌĆö `cast code` returns empty) |
| `VAULT.admin()`                    | `0xf39Fd6e51aad88F6F4ce6aB8827279cFFFb92266` (Anvil deployer) |
| Current `GDT`                      | `0x8f86403a4de0bb5791fa46b8e795c547942fe4cf` (live) |
| `PERP` (env: `PerpEngine`)         | `0x172076e0166d1f9cc711c77adf8488051744980c` |

`MarginVault.collateral` is `immutable` (declared
`IMarginToken public immutable collateral;` at the top of
`src/perps/MarginVault.sol`), so it CANNOT be changed in-place ŌĆö the
vault must be redeployed against the current GDT. Because `PerpEngine`
holds a reference to `MarginVault` in its constructor, redeploying the
vault forces a redeploy of `PerpEngine` too.

A separate diagnosis bug surfaced: the verification script
`scripts/verify-onchain-integration.sh` calls
`VAULT.collateralToken()` (which does not exist on `MarginVault`); the
correct getter is `VAULT.collateral()`. That script bug must also be
fixed in this task to unblock the GoodPerps row in the report.

The existing `script/DeployPerps.s.sol` is already env-parametrized:
```
GD_TOKEN     ŌĆö defaults to 0x36C02dA8ŌĆ”  ŌåÉ outdated
FEE_SPLITTER ŌĆö defaults to 0x3abBBŌĆ”    ŌåÉ outdated
```
Setting `GD_TOKEN=$GDT` and `FEE_SPLITTER=$FEE_SPLITTER` (from
`addresses.env`) re-runs the full Perps deploy chain (Oracle, FundingRate,
MarginVault, PerpEngine + market seeding) wired to the live GDT and
the now-correct splitter (after task 0010).

## Implementation Notes

- The `IMarginToken` interface in `src/perps/MarginVault.sol` is
  minimal (`balanceOf`, `transfer`, `transferFrom`, `approve`,
  `decimals`) ŌĆö current GDT (`GoodDollarToken.sol`) implements all of
  these via standard ERC-20.
- After the redeploy, the old `MarginVault` and `PerpEngine` will be
  orphaned but their bytecode stays on chain (Anvil doesn't garbage-
  collect). They will simply no longer be referenced by `addresses.env`
  or the integration script.
- `DeployPerps.s.sol` already calls `seedMarkets()` (or its inline
  equivalent) for 6 markets ŌĆö we will reuse that exactly. Do NOT
  duplicate market seeding.
- The shell-script fix is one-line in
  `scripts/verify-onchain-integration.sh`:
  ```diff
  - vault_collat=$(cast call "$VAULT" "collateralToken()(address)" --rpc-url "$RPC" 2>/dev/null)
  + vault_collat=$(cast call "$VAULT" "collateral()(address)" --rpc-url "$RPC" 2>/dev/null)
  ```
  Match the exact line by inspecting the file before editing.
- Frontend SDK references `VAULT` and `PERP` via `addresses.env` ŌåÆ
  `frontend/src/lib/addresses.ts` (regenerated by
  `scripts/refresh-frontend-addresses.py` if it exists, else manually
  by syncing the env into the TS const map). Verify whether the
  frontend has a typed addresses module before committing to confirm
  we don't break the UI; if it does, run the existing sync script.
  This task's acceptance criteria do NOT require frontend changes ŌĆö
  only that no UI regression is introduced.
